# Pocket Edition

Pocket edition runs Goload as a single local app. It is intended for a user-owned machine where the browser UI and server process are both local.

---

## What Runs In Pocket

`cmd/pocket/main.go` wires the same domain services used by microservice mode with local implementations:

| Concern | Pocket implementation |
|---------|-----------------------|
| HTTP API and web UI | One HTTP server on `HTTP_ADDRESS` |
| Auth | Default local `pocket` account with no-auth middleware |
| Task persistence | SQLite |
| Task/download events | In-memory broker |
| Download storage | Local filesystem under `POCKET_DATA_DIR` |
| Download tokens | In-memory token store |
| Frontend static files | `POCKET_WEB_DIR` |

---

## Configuration

| Environment variable | Default | Description |
|----------------------|---------|-------------|
| `HTTP_ADDRESS` | `0.0.0.0:8080` | HTTP listen address |
| `POCKET_DB_PATH` | `./goload.db` | SQLite database path |
| `POCKET_DATA_DIR` | `./data` | Local storage root |
| `POCKET_WEB_DIR` | `./public/dist` | Compiled frontend directory |
| `LOG_LEVEL` | `debug` | Log level |

The pocket Dockerfiles build the frontend with:

```bash
VITE_GOLOAD_API_URL=http://localhost:8080
VITE_GOLOAD_POCKET=true
```

`VITE_GOLOAD_POCKET=true` enables pocket-only UI behavior such as **Show in folder**.

---

## Download Storage

Pocket stores downloaded files under `POCKET_DATA_DIR` using storage keys generated by the download service:

```text
{taskID}/{safeFileName}-{sourceHash}{ext}
```

Example:

```text
data/1/design_rationale_example_1-9e04fb677787202d.pdf
```

Each stored object also has a sidecar metadata file:

```text
data/1/design_rationale_example_1-9e04fb677787202d.pdf.meta.json
```

The metadata includes the resolved filename, file size, content type, storage key, and timestamps.

## Browser Download vs Show In Folder

Browsers do not reliably allow JavaScript web apps to open `file://` URLs. Pocket therefore avoids returning local filesystem presigned URLs for completed downloads.

Pocket has two completed-file actions:

- **Show** / **Show in folder**: calls `POST /api/v1/pocket/tasks/reveal?id=<taskId>` and asks the local server to open the OS file manager at the already-stored file. This does not create a second copy.
- **Download copy**: uses `/download?token=...` and streams the stored file over local HTTP to the browser. This creates a browser download copy.

---

## Reveal Endpoint

```http
POST /api/v1/pocket/tasks/reveal?id=<taskId>
```

Response:

```json
{
  "path": "C:\\Users\\Admin\\Downloads\\pocket-release\\data\\1\\file.pdf"
}
```

Platform behavior:

| Platform | Command |
|----------|---------|
| Windows | `explorer.exe /select,"<path>"` |
| WSL | `wslpath -w <path>` then `explorer.exe /select,"<windows-path>"` |
| macOS | `open -R <path>` |
| Linux | `xdg-open <dir>` or `gio open <dir>` |

If pocket runs inside a Docker container, the container path may not match the host path. In that setup, **Show in folder** can only reveal paths visible to the process that runs pocket.

## Build

Build the pocket web stage:

```bash
docker build --target web-builder -f Dockerfile.pocket -t goload-pocket-web-check .
```

Build the release packager:

```bash
docker build -f Dockerfile.pocketbuilder -t goload-pocketbuilder .
```
